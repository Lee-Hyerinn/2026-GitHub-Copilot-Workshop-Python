# Pomodoroタイマー 段階的実装計画

## 1. 実装粒度の方針

- フェーズ: 0.5〜1週間で完了する機能群（必ず動く価値を出す）
- スライス: 1〜2日で完了する縦切り実装（UI + ロジック + 最低限テスト）
- タスク: 2〜4時間で終わる作業単位
- PR: 1スライス = 1PR（差分は小さく、目的は1つ）

## 2. フェーズ別実装計画

### フェーズ0: 土台構築

目的
- 開発を回せる最小基盤を作る

実装内容
- Flask起動
- テンプレート配信
- 静的ファイル配信
- 基本ディレクトリ構成の準備

完了条件
- 画面が表示される
- 空UIでも配信できる

テスト
- Flaskルート疎通テスト

---

### フェーズ1: UIモック（見た目固定）

目的
- 添付デザインに近い見た目と操作導線を先に確定する

実装内容
- タイマー表示（モード、残り時間）
- 円形プログレス表示
- 操作ボタン（開始 / 一時停止 / 再開 / リセット / スキップ）
- 今日の進捗カード（初期はダミーデータ可）

完了条件
- デザインレビュー可能な状態
- 主要操作導線が確認できる

テスト
- 主要UI要素の存在確認

---

### フェーズ2: Core状態機械

目的
- ドメインロジックを副作用なしで確定する

実装内容
- 状態: focus / short_break / long_break / paused
- イベント: start / pause / resume / reset / skip / tick / complete
- reducer形式で状態遷移を定義
- 不正遷移ガード（未開始でpause不可など）

完了条件
- 状態遷移表どおりに全イベントが処理される

テスト
- 状態遷移ユニットテストを最優先で網羅

---

### フェーズ3: 時間制御とController接続

目的
- 実際にカウントダウンを動かす

実装内容
- endAt基準の残時間再計算
- tick連携
- 完了イベント発火
- UI反映

完了条件
- タブ切替後も時間ズレが小さい
- 完了時に正しく次状態へ遷移する

テスト
- FakeClockで時間計算テスト
- Controllerユースケーステスト

---

### フェーズ4: ローカル永続化（MVP実用化）

目的
- 再読込後も設定と進捗を保持する

実装内容
- LocalStorageアダプタ実装
- 設定保存と復元
- 当日進捗の保持

完了条件
- リロード後に設定と進捗が再現される

テスト
- Storageアダプタ契約テスト

---

### フェーズ5: Flask設定API連携

目的
- フロント単体設定管理からAPI管理へ拡張する

実装内容
- GET /api/settings
- PUT /api/settings
- 入力バリデーション
- エラーハンドリング（400 / 500）

完了条件
- UIから設定取得・更新が往復できる

テスト
- Flask test clientでAPIテスト（正常系 / 異常系）

---

### フェーズ6: 通知・音・アクセシビリティ

目的
- 実運用で使いやすい品質に引き上げる

実装内容
- 通知許可フロー
- 完了音再生
- キーボード操作対応
- ARIAラベル整備
- モバイル表示最適化

完了条件
- キーボードのみで主要操作可能
- 通知・音が設定どおりに動作

テスト
- 手動E2Eチェックリスト
- 主要機能の自動テスト

---

### フェーズ7: 履歴機能（必要時）

目的
- 実績分析に向けたデータ蓄積を可能にする

実装内容
- POST /api/sessions
- GET /api/sessions?date=YYYY-MM-DD
- 必要に応じてSQLite導入
- 履歴テーブル設計（planned_seconds / actual_seconds / completed など）

完了条件
- セッション保存と日次取得が可能

テスト
- 履歴APIテスト
- データ整合性テスト

## 3. 各フェーズの運用ルール

- フェーズ開始前に受け入れ条件を3〜5個に固定する
- 1スライス1PRで進める
- 各フェーズでテスト追加まで完了して次に進む
- UIモックはCore導入後に必ず実ロジックへ置換する

## 4. 2週間の実行例

- 1週目前半: フェーズ0〜2（基盤、UIモック、Core）
- 1週目後半: フェーズ3（時間制御）
- 2週目: フェーズ4〜6（永続化、設定API、通知/アクセシビリティ）
- フェーズ7は要件発生時に着手
